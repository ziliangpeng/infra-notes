## Frequency Penalty Investigation

**Finding:** `frequency_penalty=1.3` default comes from:
- `chat-stack/tokenizer/utils/adapters/vllm/request.py` line 38: `_DEFAULT_FREQUENCY_PENALTY: float = 1.3`
- Applied to all vLLM requests if not set in `stream_params`
- Also in `deployments/charts/dynamic-router/templates/chat-service-dynamic-router-config.yaml`

**vLLM frequency_penalty behavior (confirmed from source):**
- Only counts **output tokens** (not prompt/input tokens)
- Formula: `logit -= freq_penalty * output_bin_counts[token]`
- 1.3 is very aggressive — after 5 occurrences of `.`, logit drops by 6.5 → causes "runaway" generation (no punctuation, continuous words)
- Original 1.3 was tuned for Mistral, not DeepSeek-V3

**Experiment arms and frequency penalties:**
| Model | freq_penalty |
|-------|-------------|
| deepseek-v3-fp8-mi350 | 1.3 (default) |
| deepseek-v3-fp8-mi350-v17 | 1.3 |
| deepseek-v3-fp8-mi350-v17-b | 1.0 |
| deepseek-v3-fp8-mi325 | 1.3 |
| deepseek-v3-fp8-mi325-v17 | 1.3 |
| deepseek-v3-fp8-mi325-v17-b | 0.8 (changed mid-experiment) |
| deepseek-v3-fp8-mi325-v17-c | 0.8 (fork of v17-b, 2 replicas) |
| deepseek-v3-fp8-mi325-v17-x | 0.8 |
| deepseek-v3-fp8-mi325-v17-y | 0.0 |
| qwen3-235b-prod-1 | 1.3 (default) |

## DeepSeek-V3 DP8 EP8 on MI325X (dev pod) — mori all2all investigation

**Date:** 2026-03-31

### Setup
- Dev pod: `deepseek-v3-fp8-mi325-v18-dpa` (ns: vllm-model-server, node: amd2-mi325x-vdgf2)
- GPU: 8x MI325X, actually reports gfx942 in rocm-smi (not gfx1100 as expected)
- vLLM: v0.18.1 ROCm image
- Config: DP8 EP8 TP1, model=DeepSeek-V3-0324 FP8

### all2all backend findings

Available backends in vLLM v0.18.1:
- `allgather_reducescatter` (default, RCCL-based, always works)
- `deepep_high_throughput`, `deepep_low_latency` — NOT installed in image
- `pplx` — NOT installed in image
- `mori` — **installed and importable** (gfx942 native AMD kernel)
- `flashinfer` — NOT installed

### mori incompatibility with FP8

mori requires AITER fused_moe (`VLLM_ROCM_USE_AITER_MOE=1`).
However, AITER MOE + FP8 quantization sets `defer_input_quant=True`, which mori does NOT support.

Error: `MoriPrepareAndFinalize does not support defer_input_quant=True. Please select an MoE kernel that accepts quantized inputs.`

This is a **dead end in v0.18.1**: mori requires AITER MOE, but AITER MOE + FP8 = incompatible with mori.

### Working config (no mori)
```bash
VLLM_ROCM_USE_AITER=1
VLLM_ROCM_USE_AITER_MOE=1
SAFETENSORS_FAST_GPU=1
VLLM_SKIP_P2P_CHECK=1

vllm serve DeepSeek-V3-0324 \
  --data-parallel-size=8 \
  --enable-expert-parallel \
  --quantization fp8 \
  --max-model-len 10000 \
  # no --all2all-backend → defaults to allgather_reducescatter
```
