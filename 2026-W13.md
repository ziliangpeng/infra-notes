## AMD2 MI350X — deepseek-v3-fp8-mi350-mtp TP4 调试 (Mar 24)

### 背景

PR #17458：把 deepseek-v3-fp8-mi350-mtp 从 TP8 1 replica 改成 TP4 2 replicas，提升并发。

### 尝试过程

**问题 1：GPU 资源不足，新 pod Pending**

TP4 rolling update 时，旧 TP8 pod 占着 8 GPU 不释放（rolling update 默认等新 pod ready 才删旧 pod）。
新 pod 一直 Pending，`cluster-autoscaler: max node group size reached`。

解决方案：手动 `kubectl delete pod` 强删旧 pod，同时 `kubectl scale rs ... --replicas=0` 阻止旧 ReplicaSet 重建。

---

**问题 2：AITER MLA kernel crash（TP4 + MTP + fp8 kv-cache 不兼容）**

错误：
```
RuntimeError: Worker failed with error 'get_heuristic_kernel_mla: cannot get heuristic kernel!
q_type:fp8 kv_type:fp8 gqa:32 ps:0 prefill:0 causal:0 qseqlen:2'
```

原因：AITER MLA kernel 不支持 fp8 kv-cache + MTP speculative decode（qseqlen=2）在 TP4 下的组合。

修复：`VLLM_ROCM_USE_AITER_MLA=0`

注意坑：deployment 启动脚本里 hardcode 了 `export VLLM_ROCM_USE_AITER_MLA="1"`，
所以 `kubectl set env` 不够用，会被脚本里的 export 覆盖。
必须直接 patch deployment 的 args（启动脚本）才有效。

命令：
```bash
kubectl patch deployment <name> -n vllm-model-server --patch-file /tmp/patch.json
```

---

**问题 3：No common block size for 1（TP4 + block-size=1 不兼容）**

错误：
```
ValueError: No common block size for 1.
RuntimeError: Worker failed with error 'No common block size for 1.'
```

出现条件：AITER_MLA=0 后仍然 crash。原因是 `--block-size=1` 在 TP4 下不支持。
TP8 可以用 block-size=1，但 TP4 不行（vLLM v0.18.0 限制）。

尝试：先去掉 MTP 和 block-size=1 验证 TP4 baseline → **成功 2/2 Ready**。

再加回 MTP + block-size=1 → 同样报 `No common block size for 1`，确认是 block-size=1 本身问题，与 MTP 无关。

## AMD2 MI350X — deepseek-v3-fp8-mi350-mtp TP4+AITER_MLA 深入调试 (Mar 25)

### 用 mi350-test dev pod 迭代

PR #17474: mi350-test 升级到 vllm/vllm-openai-rocm:v0.18.0，作为 dev box 手动迭代。
节点: amd2-mi350x-kwis3，8x MI350X GPU。

### 问题溯源

**问题 1: block-size=1 冲突的真正原因**

之前以为是 TP4 不支持 block-size=1，实际是 AITER_MLA=1 + fp8 kv-cache 同时存在时，
MLA backend (支持 block-size=1) 和 TRITON_ATTN (只支持 MultipleOf(16)) 被分配到同一个 kv_cache_group。

但后来发现 TRITON_ATTN 是 fallback，实际不是 fp8 kv 触发的。
更深的 root cause 见下。

**问题 2: AITER_MLA + fp8 kv-cache 根本不支持**

错误：
```
get_heuristic_kernel_mla: cannot get heuristic kernel!
q_type:fp8 kv_type:fp8 gqa:32 ps:0 prefill:0 causal:0 qseqlen:1
```

AITER MLA kernel 没有 fp8 q + fp8 kv 的 decode kernel（gqa=32, prefill=0）。
这跟 TP4/TP8 无关，跟 MTP 无关（qseqlen=1 也 crash，之前 qseqlen=2 是 MTP 加的）。

结论：**AITER_MLA=1 + --kv-cache-dtype fp8 在 v0.18.0 是 broken 组合。**

**当前测试（Mar 25 凌晨）**

去掉 --kv-cache-dtype fp8，保留其他：
- TP4 + AITER_MLA=1 + block-size=1 + EP + --quantization fp8（weight fp8，kv bf16）
- 进程在运行中，等结果。

**baseline 对比**

deepseek-v3-fp8-mi350 (TP8) 没有 --kv-cache-dtype fp8，所以 AITER_MLA=1 + block-size=1 能跑。
我们 mtp 模型多了 --kv-cache-dtype fp8，这是所有问题的根源。

---

### 完整尝试记录（deepseek-v3-fp8-mi350-mtp TP4 调试，Mar 24-25）

| # | 配置 | 结果 | 错误 |
|---|------|------|------|
| 1 | TP4 + AITER_MLA=1 + block-size=1 + kv fp8 + MTP | crash | `get_heuristic_kernel_mla: q_type:fp8 kv_type:fp8 qseqlen:2` |
| 2 | TP4 + AITER_MLA=0 + block-size=1 + kv fp8 + MTP | crash | `No common block size for 1` |
| 3 | TP4 + AITER_MLA=0 + 无block-size + kv fp8 + MTP | crash | `No common block size for 1`（MTP 强制需要 block-size=1）|
| 4 | TP4 + AITER_MLA=0 + 无block-size + kv fp8 + 无MTP | **成功 2/2 Ready** | — |
| 5 | TP4 + AITER_MLA=0 + block-size=1 + kv fp8 + 无MTP | crash | `No common block size for 1` |
| 6 | TP4 + AITER_MLA=1 + block-size=1 + kv fp8 + 无MTP | crash | `get_heuristic_kernel_mla: q_type:fp8 kv_type:fp8 qseqlen:1` |
| 7 | TP4 + AITER_MLA=1 + block-size=1 + 无kv fp8 + 无MTP | crash | `[AITER] hipModuleGetFunction named symbol not found: MLA_A16W16_1TG_4W_32mx1_16nx1_Coex0_Msk1_QH16` |

### 关键发现

1. **AITER_MLA + fp8 kv-cache = broken**
   - AITER MLA kernel 没有 `q_type:fp8 kv_type:fp8` 的 decode kernel
   - 无论 qseqlen=1 还是 qseqlen=2 都 crash
   - 跟 TP4/TP8/MTP 无关

2. **block-size=1 + AITER_MLA=0 也 broken**
   - `No common block size for 1` 错误
   - AITER_MLA=0 时用 TRITON_MLA，支持 MultipleOf(16)，与 block-size=1 不兼容

3. **AITER_MLA 在 v0.18.0 + DeepSeek-V3 GQA=32 下全面 broken**
   - fp8 kv: `get_heuristic_kernel_mla: q_type:fp8 kv_type:fp8 gqa:32` → 无此 kernel
   - bf16 kv: `MLA_A16W16_1TG_4W_32mx1_16nx1_Coex0_Msk1_QH16` → symbol not found
   - 根本原因：aiter MLA ASM kernel 未覆盖 GQA=32 的配置

4. **结论：TP4 + AITER_MLA=1 在 v0.18.0 下不可用**
   - 只能用 TRITON_MLA（AITER_MLA=0）
   - TRITON_MLA 要求 block-size 是 MultipleOf(16)，与 MTP 的 block-size=1 冲突
   - 所以 TP4 + MTP 在 v0.18.0 也不可用
   - 可行配置：TP4 + AITER_MLA=0 + 无 block-size=1 + 无 MTP（已验证 Ready）

4. **No common block size 的真正来源**
   - 源码：`vllm/v1/worker/utils.py:select_common_block_size`
   - 当多个 backend 共享一个 kv_cache_group 时，需要找公共 block size
   - rocm_aiter_mla → [1], triton_mla → [MultipleOf(16)]，无交集
