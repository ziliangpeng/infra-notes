## AMD2 MI350X — DeepSeek-V3-0324 推理实验 (Mar 23–24)

### 背景

amd2 上有 3 台 MI350X 节点（amd2-mi350x-k9p3o/k9p3x/kwis3），之前只有 mi350-test 在跑。
目标：在 MI350X 上跑 DeepSeek-V3-0324（671B），对比 MI325X（psqueak）的性能。

### 部署 1: deepseek-v3-fp8-mi350（baseline）

PR #17441, #17443 merged to main.

Config 要点：
- Image: `vllm/vllm-openai-rocm:v0.18.0`（公开镜像）
- TP=8, blockSize=1, maxModelLen=20000
- nodeSelector: amd2-mi350x-k9p3x
- extraArgs: --enable-expert-parallel --async-scheduling
- 去掉了 `-O '{"custom_ops":["-rms_norm"]}'`（公开镜像不支持）
- enableLogging: true + --no-enable-log-requests（绕过 template 的 --disable-log-request 兼容问题）

坑：
- 公开 vllm 镜像不支持 `-O '{...}'` JSON 语法（character fork 专有），导致 CrashLoopBackOff
- `--disable-log-request` 是 chart template 自动加的旧 flag，v0.18.0 已改名；用 enableLogging: true 阻止 template 加，自己加 --no-enable-log-requests

模型加载时间：约 1298 秒（~21 分钟）

Smoke test 结果（单 replica，dsv3_psq_sydney，ratio 1.0 0 0...）：
- TTFT: mean=772ms, p50=581ms, p90=1654ms
- TPOT: mean=38ms, p50=34ms
- QPS: 3.66，1317 requests，0 failures

### 部署 2: deepseek-v3-fp8-mi350-mtp（MTP Speculative Decoding）

PR #17445, #17446, #17447 merged to main.

Fork of deepseek-v3-fp8-mi350，加了：
- `--speculative-config '{"method":"mtp","num_speculative_tokens":1}'`
- 去掉 nodeSelector（允许调度到任意 MI350X 节点）

调度到 amd2-mi350x-k9p3o（空闲节点）。
模型加载时间：约 1287 秒主模型 + 96 秒 MTP head。

vLLM 日志确认 MTP 已启用：
- `Resolved architecture: DeepSeekMTPModel`
- `speculative_config=SpeculativeConfig(method='mtp', num_spec_tokens=1)`

注：`deepseek_mtp` 是旧写法，v0.18.0 里 deprecated，自动转成 `mtp`。

Replay 对比测试进行中（ratio 1.0 0.2 0...，warmup 10min，run 30min）。

### 调查过程记录

**lmserve CLI（gcp5）**

路径：`/home/common/git-sync/k8s/k8s.git/bin/lmserve`
本地 alias `scale-amd2` 定义在 `~/Dropbox/config/localconfig/shared/cai.zsh`，封装了：
```
lmserve scale -c amd2 -r $replicas -m $deployment --skip-capacity-check
```
常用命令：
- `lmserve scale -c amd2 -r 1 -m <manifest> --skip-capacity-check`
- `lmserve sync --manifest <name> --cluster amd2 --skip-capacity-check`
- `lmserve manifest status --manifest <name> --last 500`

**vLLM 版本调查（upstream repo）**

克隆了 vllm upstream 到 /tmp/vllm-full，对比 v0.15.0 和 v0.18.0：
- `-O '{...}'` JSON 语法是 character fork 专有，upstream 的 `-O` 只接受数字（0/1/2/3）映射到 `--optimization-level`
- `--disable-log-request` 在 v0.18.0 改名为 `--no-enable-log-requests`
- `deepseek_mtp` method 在 v0.18.0 deprecated，代码里自动转成 `mtp`

**chart template 的坑**

`charts/vllm-model-server/templates/deployment.yaml` 第 384 行：
```
{{- if not .enableLogging }}
CMD+=(--disable-log-request)
{{- end }}
```
解决方案：`enableLogging: true` 阻止 template 加旧 flag，自己在 extraArgs 加 `--no-enable-log-requests`

**AMD2 MI350X 节点状态**

3 台节点（192 CPU，~2TB host memory，8x GPU 288GB each）：
- `amd2-mi350x-k9p3x` → deepseek-v3-fp8-mi350（baseline）
- `amd2-mi350x-k9p3o` → deepseek-v3-fp8-mi350-mtp（MTP）
- `amd2-mi350x-kwis3` → mi350-test（其他人的）

**Config 优化讨论结论**

排除的优化项：
- cpusPerReplica → CPU 是 request 不是 limit，不影响性能
- memoryPerReplica → 不用于 KV cache，不影响性能
- VLLM_ROCM_USE_AITER_MHA → DeepSeek 用 MLA 不是 MHA，无效
- --enable-chunked-prefill → VLLM_USE_V1=1 时默认已开启
- --max-num-seqs → replay in-flight 峰值才 90，默认 256 够用

潜在优化项（未测试）：
- 去掉 NCCL_MIN_NCHANNELS: 112（MI325X 调优值，MI350X 可能不适用）
- 去掉 VLLM_ROCM_QUICK_REDUCE_QUANTIZATION: INT6（公开镜像可能不支持）

**Replay 结果（baseline，dsv3_psq_sydney）**

ratio=1.0 0 0...，warmup=60s，duration=300s：
- 1317 requests，0 failures，avg QPS=3.66
- TTFT: mean=772ms, p50=581ms, p75=995ms, p90=1654ms, p95=2061ms
- TPOT: mean=38ms, p50=34ms, p90=64ms

ratio=1.0 0.3 0...（pod1 full + pod2 30%），warmup=60s，duration=300s：

**Port-forward 问题**

本地 `kubectl port-forward` 长时间高并发下会断（Too many open files，Errno 24）。
解决方案：使用外部 URL `https://amd2.charactertech.io/vllm/models/<model>/v1/completions`，无需 port-forward。
