## AMD2 MI350X — deepseek-v3-fp8-mi350-mtp TP4+AITER_MLA 深入调试 (Mar 25)

### 用 mi350-test dev pod 迭代

PR #17474: mi350-test 升级到 vllm/vllm-openai-rocm:v0.18.0，作为 dev box 手动迭代。
节点: amd2-mi350x-kwis3，8x MI350X GPU。

### 问题溯源

**问题 1: block-size=1 冲突的真正原因**

之前以为是 TP4 不支持 block-size=1，实际是 AITER_MLA=1 + fp8 kv-cache 同时存在时，
MLA backend (支持 block-size=1) 和 TRITON_ATTN (只支持 MultipleOf(16)) 被分配到同一个 kv_cache_group。

但后来发现 TRITON_ATTN 是 fallback，实际不是 fp8 kv 触发的。
更深的 root cause 见下。

**问题 2: AITER_MLA + fp8 kv-cache 根本不支持**

错误：
```
get_heuristic_kernel_mla: cannot get heuristic kernel!
q_type:fp8 kv_type:fp8 gqa:32 ps:0 prefill:0 causal:0 qseqlen:1
```

AITER MLA kernel 没有 fp8 q + fp8 kv 的 decode kernel（gqa=32, prefill=0）。
这跟 TP4/TP8 无关，跟 MTP 无关（qseqlen=1 也 crash，之前 qseqlen=2 是 MTP 加的）。

结论：**AITER_MLA=1 + --kv-cache-dtype fp8 在 v0.18.0 是 broken 组合。**

**当前测试（Mar 25 凌晨）**

去掉 --kv-cache-dtype fp8，保留其他：
- TP4 + AITER_MLA=1 + block-size=1 + EP + --quantization fp8（weight fp8，kv bf16）
- 进程在运行中，等结果。

**baseline 对比**

deepseek-v3-fp8-mi350 (TP8) 没有 --kv-cache-dtype fp8，所以 AITER_MLA=1 + block-size=1 能跑。
我们 mtp 模型多了 --kv-cache-dtype fp8，这是所有问题的根源。

### --profile flag 实现（Mar 25）

新增 `--profile` CLI flag，使用 CUDA events 分别计时：
- `profile/fwd_attn_ms`：forward attention 时间
- `profile/fwd_other_ms`：forward 其余时间
- `profile/bwd_attn_ms`：backward attention 时间（从 fwd 比例估算）
- `profile/bwd_other_ms`：backward 其余时间
- `profile/attn_pct`：attention 占总时间百分比

实现坑：
1. `torch.compile` 下 CUDA events 在 forward 里不生效（dynamo inline module calls）
2. backward module hooks 在 compile 后也不触发
3. 解法：`--profile` 模式下自动禁用 `torch.compile`，牺牲速度换准确性

profile 结果（无 torch.compile，MI325X，LLaMA 1B）：
- fwd attn=43.3ms  fwd other=630.7ms
- bwd attn=46.8ms  bwd other=681.1ms
- attention 占总时间 **6.4%**（其实非常小！）
- 说明瓶颈不在 attention，而在 Linear（FFN/QKV projection）

结论：优化 attention kernel 对 LLaMA 1B 帮助有限（attention 只占 6%），
主要瓶颈是 GEMM（FFN SwiGLU 和 QKV projections）。

代码位置：~/training/20260325-llama-1b-mi325-aiter/train.py
用法：torchrun --nproc_per_node=1 train.py --tokens 1 --profile [--prod]

| 4 | TP4 + AITER_MLA=0 + 无block-size + kv fp8 + 无MTP | **成功 2/2 Ready** | — |

### 关键发现

1. **AITER_MLA + fp8 kv-cache = broken**
   - AITER MLA kernel 没有 `q_type:fp8 kv_type:fp8` 的 decode kernel
   - 无论 qseqlen=1 还是 qseqlen=2 都 crash
   - 跟 TP4/TP8/MTP 无关

2. **block-size=1 + AITER_MLA=0 也 broken**
   - `No common block size for 1` 错误
   - AITER_MLA=0 时用 TRITON_MLA，支持 MultipleOf(16)，与 block-size=1 不兼容

3. **AITER_MLA 在 v0.18.0 + DeepSeek-V3 GQA=32 下全面 broken**
   - fp8 kv: `get_heuristic_kernel_mla: q_type:fp8 kv_type:fp8 gqa:32` → 无此 kernel
   - bf16 kv: `MLA_A16W16_1TG_4W_32mx1_16nx1_Coex0_Msk1_QH16` → symbol not found
   - 根本原因：aiter MLA ASM kernel 未覆盖 GQA=32 的配置

4. **结论：TP4 + AITER_MLA=1 在 v0.18.0 下不可用**
   - 只能用 TRITON_MLA（AITER_MLA=0）
   - TRITON_MLA 要求 block-size 是 MultipleOf(16)，与 MTP 的 block-size=1 冲突
   - 所以 TP4 + MTP 在 v0.18.0 也不可用
   - 可行配置：TP4 + AITER_MLA=0 + 无 block-size=1 + 无 MTP（已验证 Ready）

4. **No common block size 的真正来源**
   - 源码：`vllm/v1/worker/utils.py:select_common_block_size`
   - 当多个 backend 共享一个 kv_cache_group 时，需要找公共 block size
   - rocm_aiter_mla → [1], triton_mla → [MultipleOf(16)]，无交集

### 上游 issue 和 PR 追踪

搜索后确认我们遇到的都是已知 bug：

**Bug 1: AITER MLA 在 gfx950 (MI350X) 上 symbol not found**

- vllm issue: https://github.com/vllm-project/vllm/issues/16801
  `[BUG] ROCm: vllm fails on gfx950 with AITER MLA: named symbol not found`
- aiter issue: https://github.com/ROCm/aiter/issues/312
  `MLA kernels not compiled for gfx950 - hipModuleGetFunction fails`
- aiter tracking: https://github.com/ROCm/aiter/issues/298
  `Add gfx950 support for MLA Triton kernels`
- vllm 临时修复 PR: https://github.com/vllm-project/vllm/pull/16920
  `[ROCm] Disable AITER_MLA for gfx950 until kernels are available`

根本原因：aiter 包的 MLA kernel 只预编译了 gfx942（MI300X）和 gfx90a（MI250X），
未覆盖 gfx950（MI350X）。kernel 命名 `mla_a16w16_QH16_gqa32_hdim512_BN32_D16_stages2_bf16`
中各字段含义：
- a16w16: activation bf16, weight bf16
- QH16: 每个 TP rank 16 个 query heads
- gqa32: GQA groups = 32（DeepSeek-V3 的 kv heads 数）
- hdim512: head dim = 512
- BN32/D16: block/D dimension

Fix 状态：aiter 计划在 0.1.4 或 0.2.0 中加入 gfx950 支持，PR 在 MI350X 上测试中。
v0.18.0 里还没有修复。

**Bug 2: AITER MLA + block_size=1 + TP>=4 输出错误**

- vllm issue: https://github.com/vllm-project/vllm/issues/16900
  `[BUG] ROCm: AITER-MLA has TP4 bug with block_size=1`
- vllm fix PR: https://github.com/vllm-project/vllm/pull/16915
  `[ROCm] Fix AITER-MLA TP block_size=1 bug` — CI passing，已 approved

根本原因：AITER MLA decode 用 block_size=1（每个 block 存 1 个 token）。
TP>=4 时，block table 没有正确按 TP rank 分片，导致各 rank 访问错误的 block，
输出错误/NaN。修复方法：rank r 只使用 blocks[r*N:(r+1)*N]。

Fix 状态：PR #16915 已 approved，CI passing，未合入 v0.18.0。

**Bug 3: AITER MLA + TP=8 + gfx950 NaN**

- vllm issue: https://github.com/vllm-project/vllm/issues/16885
  `[BUG] ROCm: Inference Error with DeepSeek-V3/R1 on gfx950 using AITER MLA`

Bug 1 + Bug 2 的组合效果。

### 结论

我们在 v0.18.0 上遇到的所有 AITER MLA 问题都是上游已知 bug：
1. gfx950 上 AITER MLA kernel 根本没编译 → AITER_MLA=0 是唯一 workaround
2. 即使 gfx942 上 AITER_MLA=1 + TP4，block_size=1 也会输出错误（PR #16915 修复中）

**当前可用配置（已验证）：**
TP4 + AITER_MLA=0 + 无 block-size=1 + 无 MTP + kv fp8 → 2/2 Ready

**等待修复：**
- aiter 新版本支持 gfx950 MLA kernel → 才能开启 AITER_MLA=1
- vllm PR #16915 合入 → 才能安全使用 TP4 + AITER_MLA=1 + block_size=1 + MTP
