### DP Attention + Routing Key — Design Notes

**Goal:** Per-session KV cache affinity (same `chat_id` → same DP rank).

**Discovery:** `--schedule-policy routing-key` only prioritizes requests *within* a
single DP rank — does NOT control which DP rank a request goes to.

**Correct approach:** Set `data_parallel_rank` field in the request JSON body.
The DP controller's `maybe_external_dp_rank_routing()` routes directly if this is set.

**Implementation in prompt_replay.py:**
- Added `--dp-size N` flag
- Computes `data_parallel_rank = hash(chat_id) % dp_size` per request
- Ensures same chat always hits the same DP rank → KV cache hits

**Note on flag names:** Must use long form: `--tp-size`, `--dp-size`, `--ep-size`
(not `--tp`, `--dp`, `--ep` — those are not recognized).

and EP expert all-to-all routing when both compete for ROCm HSA resources simultaneously.

#### MORI A2A Backend — What It Is
MORI is AMD's proprietary A2A communication backend for EP on ROCm, equivalent
to NVIDIA's DeepEP. Pre-installed at /sgl-workspace/mori/ in the AMD images.

MORI supports two modes (via --deepep-mode flag, not --moe-a2a-backend):
  --deepep-mode normal         => standard MORI intra/inter-node
  --deepep-mode low_latency    => async low-latency kernel
  --deepep-mode auto           => coming soon

Two-Batch Overlapping (TBO) added in PR #17953 (Feb 20, 2026), accuracy
fix in PR #19888 (Mar 18, 2026).

Note: We tried --moe-a2a-backend mori but that may be the wrong flag.
The correct flag for MORI may be --deepep-mode normal (treating MORI as
the deepep equivalent on AMD). Worth trying.

### FIX: --cuda-graph-bs $(seq 1 256) resolves the crash

**Result: STABLE — 80/80 requests OK, zero HSA errors**

Adding `--cuda-graph-bs $(seq 1 256)` to the DP8+EP8 launch command completely
resolved the HSA_STATUS_ERROR_EXCEPTION crash. This confirms PR #20399 is the fix.

Root cause: SGLang by default includes batch sizes > 256 in cuda graph capture.
When dp-attention is enabled, mla_decode_fwd incorrectly populated kernel metadata
for those larger batch sizes, causing a GPU hardware exception under load.
Limiting cuda-graph-bs to max 256 avoids the broken code path.

```bash
  --dp-size 8 \
  --enable-dp-attention \
  --ep-size 8 \
  --cuda-graph-bs $(seq 1 256) \
  --trust-remote-code \
  --host 0.0.0.0 \
  --port 30000 \
```

### Web Research: Gloo TCP Disconnect Root Cause & Fixes

#### Root cause (confirmed from source code)
In `scheduler_dp_attn_mixin.py -> prepare_mlp_sync_batch_raw()`:

```python
if disable_overlap_schedule or SGLANG_NCCL_ALL_GATHER_IN_OVERLAP_SCHEDULER_SYNC_BATCH:
    group = tp_group.device_group   # NCCL (GPU tensors)
    device = tp_group.device
else:
    group = tp_group.cpu_group      # Gloo TCP (default in overlap mode)
    device = "cpu"
```

Default path uses `tp_group.cpu_group` = Gloo TCP backend. Under high concurrency,
the Gloo TCP connections between DP ranks get dropped → crash.

#### Known fix: PR #18386 (merged Feb 7, 2026)
Title: "fix: respect dist_timeout option"
The `--dist-timeout` server arg was NOT being passed to Gloo group init.
Gloo was using its own internal default timeout, causing connections to drop.

**Fix: pass `--dist-timeout 7200` (or higher) to prevent Gloo timeout.**

#### Env var to switch DP sync from Gloo → NCCL
```bash
export SGLANG_NCCL_ALL_GATHER_IN_OVERLAP_SCHEDULER_SYNC_BATCH=1
```
Routes the DP attention batch all_gather through NCCL (GPU) instead of Gloo (CPU TCP).
This is the cleanest workaround without changing scheduling behavior.

#### Why --disable-overlap-schedule didn't help
When overlap scheduler is disabled, code SHOULD use `device_group` (NCCL).
But the crash still happened — suggesting either:
1. There's another broadcast_pyobj code path still using Gloo, OR
2. Our image (v0.5.9, Feb 24) doesn't include PR #18386 (merged Feb 7 — should be in it?)
   or the fix doesn't fully apply to the AMD/ROCm build

#### Other relevant env vars

```bash
GLOO_SOCKET_IFNAME=eth0          # force Gloo to use specific network interface
GLOO_TIMEOUT_MS=3600000          # Gloo operation timeout in ms
```

```
a hardware exception. code: 0x1016

[DP4 TP4 EP4] Scheduler hit an exception (+ DP2, DP1)
ProcessGroupNCCL / TCPStore "Connection reset by peer" (secondary cascade)
NCCL HeartbeatMonitor detected TCPStore server shutdown
```

**Key finding:** NCCL env var WORKED for the Gloo problem — no more Gloo TCP disconnects.
BUT a new crash appeared: `HSA_STATUS_ERROR_EXCEPTION 0x1016` (GPU hardware exception).
This is the same HSA error we saw in the very first DP8+EP8 runs, BEFORE we added `--cuda-graph-bs $(seq 1 256)`.

The RCCL TCPStore "connection reset" errors are secondary cascade from the GPU crash,
NOT the primary cause.

**Revised picture:** There are TWO distinct bugs:
1. **Gloo TCP disconnect** — fixed by `SGLANG_NCCL_ALL_GATHER_IN_OVERLAP_SCHEDULER_SYNC_BATCH=1`
2. **HSA hardware exception 0x1016** — GPU kernel fault at c=16 with EP8 under load, NOT fixed yet

The HSA crash may be the same cuda-graph-bs > 256 bug (PR #20399) but triggered by a
different code path when NCCL is doing the all_gather (different memory access patterns),
or it could be a separate kernel-level EP dispatch bug.

**Comparison table:**

| Config | Crash type | Crash level |
|--------|-----------|-------------|
| DP8-only, no fixes | Gloo TCP | c=8 |
| DP8+EP8, no fixes | Gloo TCP | c=16 |
| DP8+EP8, cgbs<=256 | Gloo TCP | c=16 |
| DP8+EP8, cgbs<=256 + --disable-overlap-schedule | Gloo TCP | c=16 |
| DP8+EP8, cgbs<=256 + NCCL env var | **HSA exception 0x1016** | c=16 |
