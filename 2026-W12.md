### DP Attention + Routing Key — Design Notes

**Goal:** Per-session KV cache affinity (same `chat_id` → same DP rank).

**Discovery:** `--schedule-policy routing-key` only prioritizes requests *within* a
single DP rank — does NOT control which DP rank a request goes to.

**Correct approach:** Set `data_parallel_rank` field in the request JSON body.
The DP controller's `maybe_external_dp_rank_routing()` routes directly if this is set.

**Implementation in prompt_replay.py:**
- Added `--dp-size N` flag
- Computes `data_parallel_rank = hash(chat_id) % dp_size` per request
- Ensures same chat always hits the same DP rank → KV cache hits

**Note on flag names:** Must use long form: `--tp-size`, `--dp-size`, `--ep-size`
(not `--tp`, `--dp`, `--ep` — those are not recognized).

### Isolating DP8 vs EP8 — Which Causes the Crash?

Ran two sequential tests to isolate the root cause.

#### Test 1: DP8 only (no EP)

**Command:**
```bash
PYTHONPATH=/sgl-workspace/aiter:$PYTHONPATH nohup python3 -m sglang.launch_server \
  --model-path /scratch/DeepSeek-V3-Quark-MXFP4-v3 \
  --tp-size 8 \
  --dp-size 8 \
  --enable-dp-attention \
  --trust-remote-code \
  --host 0.0.0.0 --port 30000 --mem-fraction-static 0.85 \
  > /tmp/sglang_dp8_noep.log 2>&1 &
```

**Result: STABLE**
- Fired up in ~3 min (weights cached in RAM)
- Load test: 80/80 requests OK (40 concurrent workers)
- No HSA crashes

#### Test 2: EP8 only (no DP attention)

**Command:**
```bash
PYTHONPATH=/sgl-workspace/aiter:$PYTHONPATH nohup python3 -m sglang.launch_server \
  --model-path /scratch/DeepSeek-V3-Quark-MXFP4-v3 \
  --tp-size 8 \
  --ep-size 8 \
  --trust-remote-code \
  --host 0.0.0.0 --port 30000 --mem-fraction-static 0.85 \
  > /tmp/sglang_ep8_nodp.log 2>&1 &
```

**Result: STABLE**
- Fired up in ~24 min (each EP worker loaded weights independently — slower than DP8)
- Load test: 80/80 requests OK (40 concurrent workers)
- No HSA crashes

#### Conclusion

**Neither DP8 alone nor EP8 alone crashes. The crash is caused by DP8 + EP8 combined.**

The `HSA_STATUS_ERROR_EXCEPTION code: 0x1016` only occurs when both
`--dp-size 8 --enable-dp-attention` AND `--ep-size 8` are used together.
Likely a memory access conflict or race condition between DP attention dispatch
and EP expert all-to-all routing when both compete for ROCm HSA resources simultaneously.

#### Summary Table

| Config | Stable under load? |
|--------|--------------------|
| TP=8 only | YES |
| TP=8 + DP=8 + dp-attention (no EP) | YES |
| TP=8 + EP=8 (no DP) | YES |
| TP=8 + DP=8 + dp-attention + EP=8 | NO — HSA crash code 0x1016 |

#### MORI A2A Backend — What It Is
MORI is AMD's proprietary A2A communication backend for EP on ROCm, equivalent
to NVIDIA's DeepEP. Pre-installed at /sgl-workspace/mori/ in the AMD images.

MORI supports two modes (via --deepep-mode flag, not --moe-a2a-backend):
  --deepep-mode normal         => standard MORI intra/inter-node
  --deepep-mode low_latency    => async low-latency kernel
  --deepep-mode auto           => coming soon

Two-Batch Overlapping (TBO) added in PR #17953 (Feb 20, 2026), accuracy
fix in PR #19888 (Mar 18, 2026).

Note: We tried --moe-a2a-backend mori but that may be the wrong flag.
The correct flag for MORI may be --deepep-mode normal (treating MORI as
the deepep equivalent on AMD). Worth trying.

### FIX: --cuda-graph-bs $(seq 1 256) resolves the crash

**Result: STABLE — 80/80 requests OK, zero HSA errors**

Adding `--cuda-graph-bs $(seq 1 256)` to the DP8+EP8 launch command completely
resolved the HSA_STATUS_ERROR_EXCEPTION crash. This confirms PR #20399 is the fix.

Root cause: SGLang by default includes batch sizes > 256 in cuda graph capture.
When dp-attention is enabled, mla_decode_fwd incorrectly populated kernel metadata
for those larger batch sizes, causing a GPU hardware exception under load.
Limiting cuda-graph-bs to max 256 avoids the broken code path.

**Working DP8+EP8 launch command:**
```bash
PYTHONPATH=/sgl-workspace/aiter:$PYTHONPATH nohup python3 -m sglang.launch_server \
  --model-path /scratch/DeepSeek-V3-Quark-MXFP4-v3 \
  --tp-size 8 \
  --dp-size 8 \
  --enable-dp-attention \
  --ep-size 8 \
  --cuda-graph-bs $(seq 1 256) \
  --trust-remote-code \
  --host 0.0.0.0 \
  --port 30000 \
  --mem-fraction-static 0.85 \
  > /tmp/sglang_dp8ep8_cgbs256.log 2>&1 &
```

Log: /tmp/sglang_dp8ep8_cgbs256.log
Fired up at: 2026-03-22 23:28:18 UTC (~26 min startup)
Load test: 80/80 OK, 40 concurrent workers, no crashes

**Final stability summary:**

| Config | cuda-graph-bs | Stable? |
|--------|--------------|---------|
| TP=8 only | default | YES |
| DP8+EP8 | default (includes >256) | NO — HSA crash 0x1016 |
| DP8+EP8 | $(seq 1 256) | YES — 80/80 OK |

### Replay at 0.25 pod ratio — new crash (Gloo TCP disconnect)

**Command:**
```bash
python3 prompt_replay.py \
  --dataset dsv3_psq_sydney \
  --keep-percent-per-pod 0.25 0 0 0 0 0 0 0 \
  --model /scratch/DeepSeek-V3-Quark-MXFP4-v3 \
  --host http://localhost:30000 \
  --source-model-profile deepseek \
  --duration 300 --warmup 60 --ignore-eos --dp-size 8
```

**Trigger condition:** ~44 in-flight requests after ~41s of 1.3 QPS replay traffic.
TTFT never showed (N/A) — requests were queuing but not completing before crash.

**New crash error (different from before):**
```
RuntimeError: [/pytorch/third_party/gloo/gloo/transport/tcp/pair.cc:538]
Read error [10.150.65.174]:3989: Connection reset by peer

[DP7 TP7 EP7] Scheduler hit an exception:
  scheduler.py:1947 in get_next_batch_to_run
  -> maybe_prepare_mlp_sync_batch
  -> scheduler_dp_attn_mixin.py:255 prepare_mlp_sync_batch
```

HSA_STATUS_ERROR_EXCEPTION still appears in the log but is now a *cascade* effect
— DP7 loses its Gloo TCP connection first, then other ranks crash with HSA errors.

Root cause has shifted: the `--cuda-graph-bs $(seq 1 256)` fix resolved the
mla_decode_fwd metadata bug, but now a different issue surfaces: Gloo TCP
connections between DP ranks get disconnected under sustained real traffic load.

The crash happens inside `prepare_mlp_sync_batch` in the DP attention mixin —
this is the MLP synchronization step that coordinates between DP ranks during
the forward pass. Gloo TCP is used for this inter-rank communication.

**Revised stability table:**

| Config | Test type | Stable? |
|--------|-----------|---------|
| TP=8 only | production replay | YES |
| DP8+EP8, default cgbs | production replay | NO — HSA crash 0x1016 |
| DP8+EP8, cgbs<=256 | 80-req load test (50 tok) | YES |
| DP8+EP8, cgbs<=256 | production replay 0.25 ratio (~1.3 QPS) | NO — Gloo TCP disconnect |
