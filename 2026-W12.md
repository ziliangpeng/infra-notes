### Image 1: rocm/sgl-dev:v0.5.8.post1-rocm720-mi35x-20260219

**Status: Working (TP=8 only), crashes with DP8+EP8**

#### aiter PYTHONPATH Bug

The `aiter` package was not importable correctly out of the box.
Python finds `/sgl-workspace/aiter/` as a namespace package (the repo root)
instead of the actual package at `/sgl-workspace/aiter/aiter/`.

**Fix:** Always set `PYTHONPATH=/sgl-workspace/aiter:$PYTHONPATH` before launching.

#### Working Launch Command — TP=8 (stable baseline)

```bash
PYTHONPATH=/sgl-workspace/aiter:$PYTHONPATH nohup python3 -m sglang.launch_server \
  --model-path /scratch/DeepSeek-V3-Quark-MXFP4-v3 \
  --tp-size 8 \
  --trust-remote-code \
  --host 0.0.0.0 \
  --port 30000 \
  --mem-fraction-static 0.85 \
  > /tmp/sglang_server.log 2>&1 &
```

**Results:**
- Model loads in ~15s from /scratch (163 safetensors shards)
- Memory: ~43 GB/GPU
- Server starts: "The server is fired up and ready to roll!"
- Smoke test: all 32 requests passed, TTFT mean=364ms, 275 tok/s, 4.3 req/s

#### DP8 + EP8 Launch Command (crashes)

```bash
PYTHONPATH=/sgl-workspace/aiter:$PYTHONPATH nohup python3 -m sglang.launch_server \
  --model-path /scratch/DeepSeek-V3-Quark-MXFP4-v3 \
  --tp-size 8 \
  --dp-size 8 \
  --enable-dp-attention \
  --ep-size 8 \
  --trust-remote-code \
  --host 0.0.0.0 \
  --port 30000 \
  --mem-fraction-static 0.85 \
  > /tmp/sglang_server.log 2>&1 &
```

**Crash error:**
```
:0:rocdevice.cpp:3586: Callback: Queue 0x7f607d600000 aborting with error :
HSA_STATUS_ERROR_EXCEPTION: An HSAIL operation resulted in a hardware exception. code: 0x1016

[2026-03-22 05:25:04 DP0 TP0 EP0] Scheduler hit an exception: Traceback (most recent call last):
[2026-03-22 05:25:04 DP2 TP2 EP2] Scheduler hit an exception: Traceback (most recent call last):
```

**How to reproduce:** Start server with DP8+EP8, then send ~40+ concurrent requests
(e.g. run `prompt_replay.py` with `--keep-percent-per-pod 1.0 0 0 0 0 0 0 0`).
The server handles initial smoke test requests fine but crashes under sustained load.

**Root cause:** Default MoE all-to-all backend (`moe_a2a_backend=none`) triggers a
GPU hardware exception under EP load. Proper A2A backends (deepep, flashinfer) are
not installed in this image.

### DP Attention + Routing Key — Design Notes

**Goal:** Per-session KV cache affinity (same `chat_id` → same DP rank).

**Discovery:** `--schedule-policy routing-key` only prioritizes requests *within* a
single DP rank — does NOT control which DP rank a request goes to.

**Correct approach:** Set `data_parallel_rank` field in the request JSON body.
The DP controller's `maybe_external_dp_rank_routing()` routes directly if this is set.

**Implementation in prompt_replay.py:**
- Added `--dp-size N` flag
- Computes `data_parallel_rank = hash(chat_id) % dp_size` per request
- Ensures same chat always hits the same DP rank → KV cache hits

**Note on flag names:** Must use long form: `--tp-size`, `--dp-size`, `--ep-size`
(not `--tp`, `--dp`, `--ep` — those are not recognized).

### Isolating DP8 vs EP8 — Which Causes the Crash?

Ran two sequential tests to isolate the root cause.

#### Test 1: DP8 only (no EP)

**Command:**
```bash
PYTHONPATH=/sgl-workspace/aiter:$PYTHONPATH nohup python3 -m sglang.launch_server \
  --model-path /scratch/DeepSeek-V3-Quark-MXFP4-v3 \
  --tp-size 8 \
  --dp-size 8 \
  --enable-dp-attention \
  --trust-remote-code \
  --host 0.0.0.0 --port 30000 --mem-fraction-static 0.85 \
  > /tmp/sglang_dp8_noep.log 2>&1 &
```

**Result: STABLE**
- Fired up in ~3 min (weights cached in RAM)
- Load test: 80/80 requests OK (40 concurrent workers)
- No HSA crashes

#### Test 2: EP8 only (no DP attention)

**Command:**
```bash
PYTHONPATH=/sgl-workspace/aiter:$PYTHONPATH nohup python3 -m sglang.launch_server \
  --model-path /scratch/DeepSeek-V3-Quark-MXFP4-v3 \
  --tp-size 8 \
  --ep-size 8 \
  --trust-remote-code \
  --host 0.0.0.0 --port 30000 --mem-fraction-static 0.85 \
  > /tmp/sglang_ep8_nodp.log 2>&1 &
```

**Result: STABLE**
- Fired up in ~24 min (each EP worker loaded weights independently — slower than DP8)
- Load test: 80/80 requests OK (40 concurrent workers)
- No HSA crashes

#### Conclusion

**Neither DP8 alone nor EP8 alone crashes. The crash is caused by DP8 + EP8 combined.**

The `HSA_STATUS_ERROR_EXCEPTION code: 0x1016` only occurs when both
`--dp-size 8 --enable-dp-attention` AND `--ep-size 8` are used together.
Likely a memory access conflict or race condition between DP attention dispatch
and EP expert all-to-all routing when both compete for ROCm HSA resources simultaneously.

#### Summary Table

| Config | Stable under load? |
|--------|--------------------|
| TP=8 only | YES |
| TP=8 + DP=8 + dp-attention (no EP) | YES |
| TP=8 + EP=8 (no DP) | YES |
| TP=8 + DP=8 + dp-attention + EP=8 | NO — HSA crash code 0x1016 |
